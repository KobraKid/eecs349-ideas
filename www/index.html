    Project
    Evaluator

    Mission

    Task: Predict the success of a Kickstarter project

    Research Question: Can we predict the success of any given Kickstarter project just from the initial conditions?

    Why is this interesting?

    Kickstarter is dominated by user-generated content, and funding is driven directly by the end-users of the products. In addition, Kickstarter relies on its users for its own company's health, both on the buy and sell side. Investigating the interaction between the content of a project and the users on the platform has valeu for both Kickstarter and the users.

  • How did we achieve those results?

    Plans for remainder of the quarter (+questions/concerns)

    Currently we are in the data pre-processing stage. Our team split up into two, with one unit working on cleaning up a Kaggle dataset and the other unit working on preparing a larger dataset into a workable format. The larger dataset contains more data and more useful attributes, but it’s structured in a confusing nest of json files. We used Pandas in Python to convert the json into a dataframe, which would allow us to apply the ML techniques efficiently. However, we will have to write our own code to uncover nested information in the dataset because the dataframe seems to be only picking up the top layer of data. We will continue to munge this dataset, but the other unit on our team is using the cleaner, simpler Kaggle dataset as a smaller test case and backup.

    As expected, the majority of the time spent has been, and will be, in representing data well so that we can run algorithms on it. We expect to spend a week and half more on data pre-processing. Afterwards, we plan to use the remaining time to test our predictor with as many techniques as possible. Our final deliverable will be an analysis, using cross-validation on our data, that shows the results of our predictor using the best ML technique we tested.
    Given the significant number of categorical attributes, we will need to implement one-hot encoding to represent those attributes numerically (for compatibility with most sklearn packages). A pitfall of this approach is that it will greatly increase the number of attributes the model processes (since there will now be a new dimension for each value that all of the categorical attributes take). To address this ‘curse of dimensionality’ issue, we will analyze the frequency distributions for each categorical attribute, and one-hot encode the most common ones, while bunching the less common values into a combined category, ‘Other.’ The 80/20 rule will likely dictate our method of choosing ‘common’ categorical attribute values.
    An interesting obstacle, though perhaps not a challenging one to work around, is that some attributes only accrue values over time after a project is launched. The predictor we are building takes in a brand new project (at time t0=0 seconds) as its input, therefore it will be missing values for funding raised, total number of backers, etc. Our model will be trained on these later-occurring attributes, but it can only predict based on the attributes that would exist at t0.
